# 文档编写规范

第一版约定尽量少，但必须足够稳定，让任何人新增 Markdown 时都能知道该怎么落位。

## 新增页面的最小步骤

1. 在 `docs/` 根目录或某个子目录下新建 `*.md`。
2. 如果需要更友好的导航标题，就补一个 `title`。
3. 如果需要控制同级顺序，就补一个 `order`。

## 推荐 frontmatter

```md
---
title: 页面标题
description: 一句话摘要
order: 3
---
```

## 目录组织约定

| 场景 | 推荐位置 | 默认行为 |
| --- | --- | --- |
| 项目总览类页面 | `docs/*.md` | 进入“项目总览”分组 |
| 专题栏目首页 | `docs/<section>/index.md` | 作为该栏目的入口页 |
| 专题详情页 | `docs/<section>/*.md` | 作为栏目内页面出现在侧边栏 |

## 命名建议

- 文件名优先使用短横线风格，例如 `release-process.md`。
- 一个目录只负责一个主题，避免把不相关文档混在一起。
- 页面内使用清晰的二级、三级标题，方便右侧大纲与搜索命中。

## 图文与链接

- 相对链接优先写成 `./another-page.md` 或 `../section/page.md`。
- 相对图片优先与文档放在同目录或子目录中，便于迁移和复用。
- 需要长期复用的站点级资源可以放到 `docs/public/`。

## 当前建议

- 当前项目总览分组中优先保留关键业务输入文档，例如需求书、人员分工、实施说明或验收材料。
- 协作指南栏目只保留维护知识库所需的说明页，避免和业务资料混在一起。
