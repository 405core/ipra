## Context

当前系统前端以常规 DOM 文本方式展示大量服务端业务数据，包括旅客检索结果、旅客画像详情、管理员侧基础画像与高风险名单列表、审计日志列表与详情、辅助问询返回的策略与结论等。这种方式虽然便于开发和筛选，但浏览器可以直接读取原始文本、进行复制、打印、保存、抓包和本地缓存，不满足采购需求书第 3.2 条“数据不落地、不外流”和第 3.3 条“强制屏幕与输出水印”的要求。

本次变更是一个跨前后端、跨用户侧与管理员侧、带安全与性能约束的架构调整。核心不是前端禁止复制这么简单，而是将“所有服务端业务数据的最终呈现形态”从原始文本改造成“服务端渲染、内嵌动态水印、按场景压缩的图片资源”，同时保留按钮、输入框、筛选器、分页等站点自带交互控件的 DOM 形态，避免系统完全失去可操作性。

约束条件如下：
- 不接受“前端拿到 JSON 再自行 canvas 转图”的方案，因为这仍会把原始数据暴露给浏览器。
- 图片化后必须控制流量，不能把整个页面转成大图，也不能所有场景都无损输出。
- 现有审计日志体系已接入，需要复用并扩展为 DLP 事件留痕。
- 用户已明确要求：凡是由服务端展示的数据，而非网站本身操作控件，都应转为带水印图片。

## Goals / Non-Goals

**Goals:**
- 建立统一的“服务端业务数据图片化呈现”机制，使前端默认不再持有敏感业务文本。
- 对所有敏感展示内容强制嵌入动态水印，支持截图、拍照、录像后的泄露溯源。
- 通过分块渲染、分场景压缩、懒加载和无缓存策略，在满足安全目标的同时控制带宽消耗。
- 为用户侧、管理员侧、审计详情、辅助问询结果等核心页面提供一致的改造边界。
- 对复制、剪切、粘贴、打印、右键、拖拽等常见泄露路径进行前端交互阻断，并纳入审计。

**Non-Goals:**
- 不解决操作系统级截图、拍照、录像、OCR 的根本阻断问题；本次仅通过强水印提升追责能力。
- 不把站点所有内容都转成图片；按钮、表单、筛选器、分页、导航、弹窗控制等交互控件继续保留 DOM。
- 不在本次变更中重构全部业务 API 为全新协议；优先通过伴随式图片接口和最小文本元数据迁移。
- 不在本次变更中引入外部商业 DLP 平台、VDI 或堡垒机，只为后续终端管控预留接口与说明。

## Decisions

### 1. 采用“服务端出图，前端只展示图”的敏感数据交付模式

系统将新增一组敏感图片渲染能力，由后端在拿到真实业务数据后直接生成图片并返回前端。前端仅保留：
- 记录标识符
- 最小必要的布局元数据（宽高、渲染模式、分页信息）
- 图片地址或二进制流

这样可以避免浏览器持有完整敏感字段文本。相比“前端 JSON -> canvas”的方案，这种模式才能真正收缩浏览器数据面。

备选方案：
- 方案 A：仅禁复制、禁右键、禁选中。否决，因为浏览器仍可直接读到文本数据。
- 方案 B：前端 canvas 转图。否决，因为仍需先传输原始文本到前端。
- 方案 C：全页整屏截图输出。否决，因为不可交互、流量高、列表分页和弹窗体验差。

### 2. 采用“控件保留 DOM，服务端数据转图”的混合渲染边界

页面分为两类元素：
- 交互控件层：按钮、输入框、筛选项、分页器、导航、操作列、弹窗开关，继续保留 DOM。
- 数据展示层：所有服务端返回的业务内容，改为图片化卡片、图片化行、图片化详情块。

该边界可以满足用户提出的“凡是服务端展示的数据都转成图片”的要求，同时保留站点可用性。

### 3. 列表与详情采用不同图片粒度，不做整页大图

为降低流量与提升局部刷新能力：
- 列表页按“单条记录卡片图”或“单行记录图”输出
- 详情页按“单个详情块图”或“单份详情图”输出
- 弹窗详情单独请求，不跟列表一起下发

这样可以实现分页、懒加载和局部刷新，避免每次筛选都重新传整页图片。

### 4. 统一水印模型：动态、平铺、像素级嵌入

每张敏感图片都由后端在像素层直接烙入水印，至少包含：
- 操作员工号
- 操作员姓名
- 当前时间戳
- 客户端 IP
- 会话 ID 或请求 ID
- 页面或能力标识

水印采用多行斜向平铺，而不是单点角标。这样即使被裁剪或截图，仍有较高概率保留溯源信息。

### 5. 建立图片压缩分级策略，默认优先 WebP

为控制通信流量，后端图片输出按场景分级：
- 列表卡片：`WebP`，宽度约 900-1200，质量 70-78
- 详情卡片：`WebP`，宽度约 1200-1600，质量 78-85
- 少量需要无损文字细节的场景：降级为 `PNG`

列表图只渲染列表必需字段，详情图再输出完整内容。前端通过渲染模式参数区分 `list`、`detail`、`dialog` 等场景。这样既满足安全，又避免全量高清图导致带宽放大。

备选方案：
- 一律 PNG。否决，体积过大。
- 一律高质量 WebP。否决，列表场景浪费流量。
- 一律缩略图后点击放大。部分采用，但不能影响详情可读性，因此采用分级策略。

### 6. 前端数据流从“拉明文列表后本地过滤”改为“服务端查询 + 图片结果”

当前多个页面依赖前端 `computed` 对完整数据数组本地筛选。该模式要求前端持有完整原始数据，与本次安全目标冲突。因此改为：
- 前端只提交筛选条件
- 后端按条件查询并渲染图片结果
- 前端本地不再保存可复用的完整敏感数组

这会改变 `UserHomeView`、`UserAuditView`、`ManagementView` 以及部分 `UserAskView` 的现有状态管理方式，是本次变更的核心 breaking point。

### 7. 所有敏感图片接口默认无缓存，支持会话级短期复用

响应头统一采用：
- `Cache-Control: no-store, no-cache, must-revalidate`
- `Pragma: no-cache`

浏览器不做持久化缓存。前端如需避免同页重复请求，只允许在当前会话内以内存引用短暂复用，不写入 `localStorage`、`sessionStorage`、`IndexedDB`。

### 8. 前端 DLP 拦截与审计同步实施

虽然真正的安全边界在服务端出图，但需求书明确要求禁复制粘贴，因此前端仍需统一拦截：
- `copy`
- `cut`
- `paste`
- `contextmenu`
- `dragstart`
- `selectstart`
- `print`
- 常见快捷键：`Ctrl/Cmd + C/P/S/A`

对被拦截事件写入审计日志，形成“尝试泄露但被拦截”的留痕。

## Risks / Trade-offs

- [性能压力上升] → 服务端需要额外承担图片渲染、压缩和水印合成开销；通过缓存模板、按需渲染、限制图片尺寸与质量分级缓解。
- [列表交互复杂度上升] → 前端不能再做全量本地筛选；通过统一查询参数、分页和图片元数据协议解决。
- [可访问性下降] → 图片化会降低文本可读性和搜索能力；通过保留交互控件、控制排版密度、只对服务端数据层图片化来减轻影响。
- [无法根除截图泄露] → 系统仍可能被截图、录像或 OCR；通过强制水印、全链路审计和受控环境要求进行补偿。
- [实施范围大] → 多页面、多接口要改；通过先改高敏页面、建立统一渲染框架、逐步替换旧页面降低风险。
- [老接口兼容问题] → 现有前端依赖原始字段；通过新增伴随式图片接口和阶段性开关完成迁移，最终再清理旧文本渲染路径。

## Migration Plan

1. 先引入基础能力：
   - 服务端图片渲染工具
   - 动态水印生成器
   - 统一图片响应与压缩配置
   - 前端全局 DLP 拦截器
2. 先改最敏感和最容易落地的只读区域：
   - 用户侧日志详情
   - 管理端审计日志
   - 用户侧检索结果详情
3. 再改列表型页面：
   - 用户侧数据检索结果列表
   - 管理端基础画像、高风险名单、用户列表、日志列表
4. 最后改辅助问询返回结果：
   - 首轮策略
   - 后续追问建议
   - 总结、结论、归档信息
5. 在迁移期保留受控开关：
   - 新图片接口与旧文本接口并行
   - 页面按 capability 开关切换渲染路径
6. 回滚策略：
   - 关闭前端图片渲染开关
   - 恢复原文本 DOM 渲染路径
   - 保留 DLP 拦截和水印框架作为独立能力

## Open Questions

- 是否需要在管理员端保留少量非敏感字段明文用于快速检索提示，还是严格只保留筛选条件与记录 ID。
- 敏感图片是直接二进制流返回，还是返回短时有效的内部签名 URL；两者在日志追踪与前端复用上的权衡需要确定。
- 水印中的身份字段是否使用“姓名 + 工号”全量展示，还是按采购方要求改为警号/工号优先。
- 是否需要在部分页面加入“图片加载中占位”和失败降级策略，避免一次性切换后影响运维排障。
