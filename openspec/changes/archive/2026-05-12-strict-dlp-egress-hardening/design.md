## Context

当前系统已经有一层“受保护图片 + 前端 DLP 拦截”的基础能力，但仍然保留了大量后端到浏览器的明文数据出口，实际状态更接近“展示层部分图片化”，而不是采购需求书要求的“数据不落地、不外流”。经审查，明文暴露不仅存在于旅客检索、管理端基础画像和归档详情，也存在于辅助问询运行时、记忆服务、导入失败明细、OCR 识别结果、归档视频元数据、对象存储定位信息和旧版明文问询 / ASR 辅助接口中。

本次变更的关键不是继续补几个图片组件，而是建立一套严格的“数据出口治理模型”，以接口和数据流为单位判断是否允许明文离开后端。设计必须覆盖：
- 后端所有 HTTP / JSON / 文件 / 视频流出口
- 前端所有服务模块、页面状态和本地缓存面
- AI-Service 与后端的中转边界
- 登录、系统设置、模板下载等必要例外边界

严格口径下，“浏览器里看不到文本”并不等于达标。只要敏感文本仍以普通 JSON 进入浏览器内存、路由状态、组件状态、`localStorage`、视频元数据或对象路径，就仍然违反“数据不落地、不外流”的目标。

## Goals / Non-Goals

**Goals:**
- 为所有后端数据出口建立统一的敏感分级，并明确“禁止明文出站”的默认原则。
- 将所有服务端业务敏感数据改造为“服务端持有明文、前端只接收最小控制信息和受保护资产”的交付模式。
- 将辅助问询、问询归档、管理端列表与详情改为后端驱动，不再要求前端持有完整业务明文对象才能工作。
- 消除对象存储路径、原始转写文本、归档快照、记忆条目、导入原始行数据等侧向泄露面。
- 对保留明文的少数接口定义严格例外条件，确保例外是审计过且必要的。

**Non-Goals:**
- 不阻断操作系统级截图、拍照、录像或外部 OCR；本次仅通过内嵌水印和审计提升溯源能力。
- 不取消所有前端输入能力；用户主动录入内容、登录凭据、表单编辑态仍需以明文在浏览器中存在。
- 不将系统配置、导航标题、非业务 UI 文案等站点自带内容全部图片化。
- 不替换后端与 AI-Service 之间的明文协议；本次仅收口“后端到浏览器”的暴露面，并限制前端向后端上传的敏感明文范围。

## Decisions

### 1. 引入“后端数据出口分级矩阵”，而不是按页面零散修补

所有数据出口按以下四类处理：

- `A 类：必须明文保留`
  - 登录请求与登录响应中的 token
  - `/api/auth/me` 当前用户最小身份信息
  - 非敏感系统设置，如总轮次上限
  - 用户主动选择下载的空白模板文件
  - 用户主动录入并提交到后端的输入值
- `B 类：可返回最小控制明文`
  - 记录 ID
  - 动作可用性
  - 分页信息
  - 受保护资产引用
  - 不泄露业务内容的状态码/阶段码
- `C 类：必须改为受保护资产`
  - 所有旅客、名单、实名用户、审计、问询、归档、记忆、导入错误详情、OCR 识别文本
- `D 类：必须完全移除或重构`
  - 仅为了旧前端本地过滤/本地拼装详情而存在的明文列表接口
  - 含对象存储路径、内部桶名、原始快照 JSON 的前端消费接口

这样做的原因是页面导向的梳理容易漏掉“同一个接口被多个页面复用”或“同一份敏感数据以辅助状态再次泄露”的问题。出口矩阵以路由和 payload 为中心，更适合本次全量收口。

### 2. 采用“后端持有会话态，前端只发控制意图”的辅助问询运行模型

当前 `UserAskView` 的核心问题不是只渲染了少量明文，而是前端持有：
- 明文旅客画像
- 明文记忆上下文
- 明文问答历史
- 明文 ASR 文本
- 明文辅助判断摘要
- 明文归档快照

新的设计要求辅助问询采用“后端驱动的受保护运行态”：
- 首次进入时，前端只携带目标记录 ID 或证件号
- 后端根据服务器内画像与名单数据初始化受保护问询会话
- 前端只接收：
  - 受保护画像块
  - 受保护策略块
  - 轮次句柄
  - 动作许可
  - 少量非敏感状态
- 追问、摘要、判断、记忆刷新都由后端依据服务器会话态推进
- 前端不再调用明文 `searchPassengerProfiles()`、`fetchMemoryContext()` 等接口

备选方案是“前端继续持有敏感对象，但不显示”，否决，因为这仍然违反“数据不落地、不外流”的目标。

### 3. 问询归档采用“双层保护”：文本受保护资产化，视频使用受控流但隐藏内部存储元数据

归档的特殊性在于它同时包含文本和媒体：
- 文本部分：旅客快照、判定理由、问题包、转写、HumanOmni 摘要、轮次总结
- 媒体部分：归档视频流

文本部分全部改为受保护资产，没有争议。视频部分不能简单转图，因此采用：
- 前端不再接收 `videoUrl`、`minioBucket`、`minioObjectKey` 等内部定位字段
- 列表与详情接口仅返回 `videoId`、播放许可、展示所需最小非敏感元数据
- 视频仍通过后端代理流式访问
- 播放层继续叠加动态水印

这样可以避免把对象存储内部结构暴露给浏览器，同时保留可回放能力。

### 4. 管理端列表从“整批拉明文 + 本地筛选”改为“服务端筛选 + 受保护结果”

基础画像、高风险名单、用户管理当前的问题是：
- 前端一次性拉取完整明文数组
- 本地 `computed` 继续筛选、映射、拼接搜索文本
- 编辑弹窗复用同一份明文对象

新设计拆成两条链路：
- 展示链路：只走受保护列表 / 受保护详情
- 编辑链路：只有在点击“编辑”或“新增”时，才向后端请求或提交当前记录所需最小明文字段

这里不要求彻底禁止明文录入，因为管理端本身承担维护职责；但必须把“展示态”和“编辑态”严格隔离，避免通过列表页一次性下发整批敏感数据。

### 5. 导入、OCR、校验失败明细也纳入敏感收口，不再视为“后台工具信息”

导入失败明细经常包含：
- 证件号
- 姓名
- 电话
- 原始行数据
- 结构化错误描述

OCR 结果包含：
- 证件号
- 姓名
- 地址
- 签发机关

这些都属于明确的个人敏感信息。设计要求：
- 导入成功/失败状态可明文给出摘要计数
- 具体失败明细必须转为受保护详情资产
- OCR 识别结果不得再以明文 JSON 长驻前端，可由后端在识别成功后直接触发受保护检索，或返回受保护识别块

### 6. 明文例外边界必须可审计、可枚举、可解释

允许保留明文的接口必须同时满足：
- 不包含旅客/问询/归档/审计等业务敏感文本
- 为完成认证、导航、设置或用户主动录入所必需
- 不携带内部存储定位信息
- 不会被前端复用成业务详情

因此以下保留为明文例外：
- `/api/login`
- `/api/auth/me`
- `/api/inquiry/settings`
- `/api/admin/settings/inquiry`
- 模板下载

以下即使目前看似“工具性数据”，也不再属于例外：
- `/api/inquiry/memory-context`
- `/api/passenger-profiles`
- `/api/admin/profiles`
- `/api/admin/watchlist`
- `/api/admin/users`
- `/api/audit-logs`
- `/api/admin/inquiry-archives`
- `/api/admin/inquiry-archives/:id`
- `/api/passenger-profiles/ocr/idcard`

### 7. 引入“受保护详情句柄”和“受保护编辑加载”而不是一次性返回全量明文

为兼顾操作性与安全性：
- 只读展示：返回 `ProtectedListItem` / `ProtectedDetailResponse` / `ProtectedRuntimeSnapshot`
- 编辑入口：返回 `EditableRecordHandle`
  - 包含记录 ID、版本号、编辑授权、最小字段集
  - 仅在用户明确进入编辑动作时请求

这样可以把“读”和“写”的权限面、数据面分开，降低明文暴露范围。

### 8. 前端状态最小化：禁止将敏感业务文本写入持久化存储和长生命周期状态

设计要求前端仅允许以下位置持有明文敏感数据：
- 当前正在输入的表单控件
- 正在上传的文件对象
- 当前录音/采样过程中尚未提交的最小文本片段，如果业务确实无法避免

禁止以下做法：
- `localStorage` / `sessionStorage` / `IndexedDB` 存储敏感业务文本
- 在页面级 `ref/computed` 中长期缓存完整画像、完整转写、完整归档详情
- 用明文敏感字段拼接本地搜索索引

必要时可引入“提交即清空”的短生命周期缓冲区，并在切页、完成上传、完成归档后立即销毁。

## Risks / Trade-offs

- [实现范围极大] → 采用路由清单驱动拆分任务，优先收口明文读取链路，再逐步优化交互体验。
- [管理端编辑体验会变复杂] → 通过“只读保护 + 按需编辑加载”缓解，而不是继续整批明文下发。
- [辅助问询前端重构成本高] → 利用现有 protected session 结构扩展，而不是另起一套新协议。
- [视频无法完全图片化] → 通过后端代理播放、隐藏对象路径和持续水印降低泄露面。
- [OCR/导入结果改造会影响流程速度] → 保留摘要级明文状态提示，只把具体敏感明细移到受保护详情。
- [明文例外边界容易漂移] → 在 specs 中将例外显式列成 requirement，并要求新增接口必须先分类再上线。

## Migration Plan

1. 建立并提交“数据出口清单”作为实施基线，标注每个接口的分级、当前风险和目标状态。
2. 先关闭前端对明文敏感接口的直接消费路径，保留接口但不再由受保护页面调用。
3. 重构辅助问询，去除前端对明文画像、明文记忆、明文转写链路的依赖。
4. 重构管理端只读展示与归档详情，将列表和详情全部切到 protected-only。
5. 为导入错误详情、OCR 结果、归档视频元数据增加受保护或最小化返回协议。
6. 对不再允许前端消费的 plain API 加入服务端弃用标记，完成页面迁移后再真正下线。
7. 回归核查所有后端路由和前端服务模块，确保新增功能未绕过出口分级矩阵。

## Open Questions

- 辅助问询实时转写是否允许在“采样进行中”以极短生命周期明文存在于内存中，再在提交后立即销毁，还是必须进一步改造成后端实时转写 + 受保护增量块。
- 管理端编辑是否需要新增单记录明文加载接口，还是沿用现有列表接口但收窄为按 ID 获取的最小字段接口。
- 归档视频播放器是否需要新增短时一次性播放票据，而不是直接使用当前按 ID 的代理流。
- OCR 成功后的最佳交互是直接自动检索受保护结果，还是同时展示一张受保护 OCR 识别确认卡片。
