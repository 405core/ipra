## Context

当前系统已经按严格 DLP 方向将大部分敏感展示收口为受保护图片，但管理员端对`基础画像`和`高风险名单`仍保留了在线新增、编辑、删除和明文详情读取链路。这与需求书中“业务数据由采购方统一提供、通过本地 Excel / CSV 导入”的业务前提不一致，也使浏览器侧继续存在“主动编辑例外”的敏感明文暴露面。

从实际业务看，`基础画像`和`高风险名单`属于参考数据，不是现场处置过程中必须在线维护的运行态数据。管理员的核心需求是导入、核查、检索和使用这些数据，而不是在系统内逐条手工修订。因此，最简单且满足需求的方案不是增加更复杂的审批、复核、软删除或版本控制，而是直接收紧为`导入式只读管理`。

## Goals / Non-Goals

**Goals:**

- 去除管理员端对`基础画像`和`高风险名单`的在线新增、编辑、删除能力
- 保留管理员端对这两类数据的受保护检索、筛选和查看能力
- 在后端明确拒绝在线维护接口访问，形成前后端一致的安全边界
- 不改动现有批量导入能力，使数据维护方式完全回归需求书定义的文件导入流程

**Non-Goals:**

- 不新增双人复核、审批流、版本管理、回收站、软删除等额外机制
- 不改动`管理用户`、`系统设置`、`审计日志`、`问询归档`等其他管理能力
- 不调整数据库结构或导入模板格式
- 不变更管理员对导入结果、受保护详情和审计日志的查看能力

## Decisions

### 1. 将敏感参考数据维护方式统一为“文件导入”，不保留在线编辑例外

原因：
- 需求书只要求导入、检索、展示，没有要求管理员在线逐条维护敏感画像和名单
- 在线编辑需要把敏感字段回填到浏览器输入框，和 DLP 目标天然冲突
- 直接取消在线维护是最小改动，且最符合“数据由采购方统一提供”的业务口径

备选方案：
- 保留编辑但增强前端禁复制与水印：不能解决浏览器中出现明文输入值的问题
- 改成双人复核编辑：复杂度高，且需求书未要求
- 改成软删除 / 逻辑失效：仍然保留了在线维护入口，超出本次最小目标

### 2. 前端移除入口，后端同时拒绝接口，防止仅靠界面隐藏

原因：
- 只删前端按钮不能阻止已有脚本、旧页面或直接接口调用
- 后端显式拒绝可以形成清晰契约，也便于审计和后续联调

备选方案：
- 仅前端隐藏：安全边界不完整
- 保留接口但不在页面使用：技术债继续存在，且难以证明“已收口”

### 3. 单条明文详情读取一并收口

原因：
- 现有编辑弹窗依赖单条详情接口回填表单
- 如果只禁写但保留明文详情读取，仍然会继续扩大敏感数据在浏览器侧的暴露
- 当前管理员已经有 protected 列表与 protected 展示，不需要额外保留这条明文查看链路

备选方案：
- 只禁新增/更新/删除，保留明文详情：无法达到最小明文暴露面

## Risks / Trade-offs

- [管理员无法在系统内快速修正单条错误数据] → 通过重新导入采购方修正文件处理，保持数据来源单一
- [已有前端或测试依赖这些接口] → 同步清理管理端按钮、弹窗、service 调用和相关测试，保证前后端一致
- [物理删除接口下线后，历史误操作修复方式变化] → 明确业务规则为“不在系统内删除敏感参考数据，只通过导入覆盖维护”
- [接口行为变化属于 breaking change] → 在 proposal 和 spec 中明确标注，并返回可读拒绝信息

## Migration Plan

1. 前端管理页移除`基础画像`和`高风险名单`的新增、编辑、删除入口与表单状态
2. 前端 service 删除不再使用的创建、更新、删除、单条明文读取调用
3. 后端将对应接口改为返回拒绝或直接移除路由注册
4. 保留并验证导入、protected 列表、protected 检索链路可正常工作
5. 更新测试，覆盖“管理端只能导入和查看，不能在线维护”的新约束

回滚策略：

- 如需临时恢复，只需恢复前端入口与后端接口注册，但这会重新引入明文编辑面，不建议作为常规回滚路径

## Open Questions

- 无。当前需求书和业务口径已经足够明确，本次按最小收口方案执行即可。
